=== Потоки данных и высокоуровневое взаимодействие

=== Основной поток обработки пользовательского запроса

[source,text]
----
1. Пользователь → Frontend
   - Пользователь отправляет запрос через веб-интерфейс
   - Frontend валидирует данные на клиенте

2. Frontend → API Gateway
   - HTTP/HTTPS запрос с данными пользователя
   - Заголовки: Authorization (JWT токен), Content-Type, Request-ID

3. API Gateway → IAM (внутренний/внешний)
   - Проверка JWT токена
   - Валидация прав доступа (OAuth2/OIDC)
   - Получение информации о пользователе

4. API Gateway → Backend Service
   - Маршрутизация запроса к соответствующему микросервису
   - Rate limiting проверка
   - Добавление контекстной информации (User-ID, Trace-ID)

5. Backend Service → Cache (Redis)
   - Проверка кэша для часто запрашиваемых данных
   - Cache-Aside: если данных нет в кэше, запрос к БД

6. Backend Service → Database
   - Запрос данных из PostgreSQL (если нет в кэше)
   - Операции записи через Write Model
   - Операции чтения через Read Replicas

7. Backend Service → External Systems (при необходимости)
   - Платежная система: синхронный вызов через REST API
   - Система уведомлений: асинхронная отправка через Event Bus

8. Backend Service → API Gateway
   - Возврат результата обработки
   - Статус код, данные ответа, метаданные

9. API Gateway → Frontend
   - Формирование HTTP ответа
   - Добавление заголовков безопасности

10. Frontend → Пользователь
    - Отображение результата
    - Обработка ошибок и уведомлений
----
=== Поток аутентификации и авторизации

[source,text]
----
1. Пользователь → Frontend
   - Ввод учетных данных (логин/пароль)

2. Frontend → API Gateway
   - POST /auth/login с учетными данными

3. API Gateway → IAM (внешний)
   - OAuth2 Authorization Code Flow
   - Проверка учетных данных
   - Генерация access token и refresh token

4. IAM → API Gateway
   - Возврат токенов (JWT)
   - Информация о пользователе и ролях

5. API Gateway → Frontend
   - Передача токенов клиенту
   - Сохранение токенов в secure storage

6. Последующие запросы:
   - Frontend добавляет Authorization header с JWT
   - API Gateway валидирует токен локально или через IAM
   - При истечении токена - обновление через refresh token
----

=== Поток обработки платежа

[source,text]
----
1. Backend Service → Payment System
   - Инициация платежа через REST API
   - Передача данных платежа (сумма, валюта, описание)
   - Circuit Breaker для защиты от сбоев

2. Payment System → Backend Service
   - Подтверждение приема платежа
   - Transaction ID для отслеживания

3. Backend Service → Database
   - Сохранение статуса платежа (Pending)
   - Outbox Pattern: сохранение события в таблицу outbox

4. Payment System → Backend Service (Webhook)
   - Асинхронное уведомление о статусе платежа
   - Callback с результатом обработки

5. Backend Service → Database
   - Обновление статуса платежа (Success/Failed)
   - Saga Pattern: компенсирующие действия при ошибке

6. Backend Service → Notification System (Event)
   - Отправка уведомления пользователю через Event Bus
   - Асинхронная обработка через Outbox Pattern
----

=== Поток отправки уведомлений

[source,text]
----
1. Backend Service → Event Bus
   - Публикация события уведомления
   - Event содержит: тип уведомления, получатель, данные

2. Event Bus → Notification Service
   - Асинхронная доставка события
   - Retry Pattern при временных сбоях

3. Notification Service → External Notification System
   - Отправка через REST API или SMTP
   - Поддержка различных каналов (SMS, Email, Push)

4. External Notification System → Notification Service
   - Статус доставки уведомления

5. Notification Service → Database
   - Сохранение статуса доставки
   - Логирование для аудита
----

=== Поток мониторинга и логирования

[source,text]
----
1. Все компоненты → Monitoring System
   - Отправка метрик (Prometheus format)
   - Health checks каждые 30 секунд
   - Distributed tracing (OpenTelemetry)

2. API Gateway → Logging System
   - Логирование всех входящих запросов
   - Структурированное логирование (JSON)
   - Correlation ID для трейсинга

3. Backend Services → Logging System
   - Логирование бизнес-событий
   - Ошибки и исключения
   - Производительность операций

4. Database → Monitoring System
   - Метрики производительности БД
   - Размер БД, количество соединений
   - Query performance metrics

5. Monitoring System → Alerting System
   - Генерация алертов при превышении порогов
   - Интеграция с системами уведомлений
----

=== Критичные компоненты и потоки данных

==== Критичные компоненты системы

*API Gateway*::
  * Критичность: КРИТИЧНО
  * Обоснование:
    ** Единая точка входа для всех запросов
    ** Отказ приводит к недоступности всей системы
    ** Обрабатывает аутентификацию и авторизацию
  * Требования к надежности:
    ** Минимум 3 инстанса с балансировкой нагрузки
    ** Health checks каждые 10 секунд
    ** Автоматическое исключение неработающих инстансов
    ** Rate limiting для защиты от перегрузки
  * Шаблоны: Circuit Breaker, Retry, Bulkhead

*Backend Services (Обработка запросов)*::
  * Критичность: КРИТИЧНО
  * Обоснование:
    ** Реализует основную бизнес-логику
    ** Обрабатывает до 10 миллионов запросов в секунду
    ** Отказ приводит к потере функциональности
  * Требования к надежности:
    ** Горизонтальное масштабирование (автоскейлинг)
    ** Минимум 5 инстансов в каждом регионе
    ** Изоляция критичных операций (Bulkhead)
    ** Graceful shutdown
  * Шаблоны: Microservices, CQRS, Cache-Aside, Connection Pooling

*Database (PostgreSQL)*::
  * Критичность: КРИТИЧНО
  * Обоснование:
    ** Хранит все данные пользователей
    ** Zero data loss requirement
    ** Отказ приводит к потере данных и недоступности
  * Требования к надежности:
    ** Master-Replica репликация (минимум 3 реплики)
    ** Автоматический failover
    ** Регулярные бэкапы (каждые 6 часов)
    ** Репликация в 3 регионах
    ** Мониторинг производительности запросов
  * Шаблоны: Read Replicas, Database per Service, Outbox Pattern

*IAM (Identity and Access Management)*::
  * Критичность: ВЫСОКАЯ
  * Обоснование:
    ** Контроль доступа ко всей системе
    ** Отказ блокирует всех пользователей
    ** Безопасность системы зависит от IAM
  * Требования к надежности:
    ** Высокая доступность (99.99%)
    ** Кэширование токенов для снижения нагрузки
    ** Резервная система IAM
  * Шаблоны: Circuit Breaker, Cache-Aside

*Cache Layer (Redis)*::
  * Критичность: ВЫСОКАЯ
  * Обоснование:
    ** Критичен для производительности
    ** Снижает нагрузку на БД на 80-90%
    ** Отказ приводит к деградации производительности
  * Требования к надежности:
    ** Redis Cluster с репликацией
    ** Автоматический failover
    ** Мониторинг hit rate
  * Шаблоны: Cache-Aside, Write-Through Cache

==== Критичные потоки данных

*Поток обработки пользовательского запроса*::
  * Критичность: КРИТИЧНО
  * Компоненты: Frontend → API Gateway → Backend → Database
  * Требования:
    ** Latency < 100ms для 95% запросов
    ** Пропускная способность: 10M RPS
    ** Zero data loss
    ** End-to-end tracing
  * Защита:
    ** Rate limiting на API Gateway
    ** Circuit Breaker для внешних зависимостей
    ** Retry с exponential backoff
    ** Bulkhead для изоляции критичных операций

*Поток аутентификации*::
  * Критичность: КРИТИЧНО
  * Компоненты: Frontend → API Gateway → IAM
  * Требования:
    ** Latency < 50ms для проверки токенов
    ** Безопасность: защита от атак
    ** Кэширование токенов для производительности
  * Защита:
    ** JWT валидация с минимальной задержкой
    ** Кэширование валидных токенов
    ** Circuit Breaker для IAM интеграции

*Поток записи данных*::
  * Критичность: КРИТИЧНО
  * Компоненты: Backend → Database (Master)
  * Требования:
    ** Консистентность данных (ACID)
    ** Zero data loss
    ** Транзакционная целостность
  * Защита:
    ** Write-Through Cache для критичных данных
    ** Outbox Pattern для событий
    ** Репликация с синхронной записью

*Поток чтения данных*::
  * Критичность: ВЫСОКАЯ
  * Компоненты: Backend → Cache → Database Replicas
  * Требования:
    ** Latency < 50ms для чтения
    ** Высокая пропускная способность
    ** Eventual consistency допустима
  * Оптимизация:
    ** Cache-Aside для частых запросов
    ** Read Replicas для распределения нагрузки
    ** Connection Pooling

*Поток обработки платежей*::
  * Критичность: КРИТИЧНО
  * Компоненты: Backend → Payment System → Database
  * Требования:
    ** Транзакционная целостность
    ** Идемпотентность операций
    ** Компенсирующие транзакции при ошибках
  * Защита:
    ** Saga Pattern для распределенных транзакций
    ** Circuit Breaker для платежной системы
    ** Outbox Pattern для событий
    ** Retry с ограничением попыток

==== Стратегия защиты критичных компонентов

*Избыточность*::
  * Все критичные компоненты развернуты минимум в 3 инстансах
  * Географическое распределение в 3 регионах
  * Автоматический failover при отказе инстанса

*Мониторинг*::
  * Health checks каждые 10 секунд для критичных компонентов
  * Метрики производительности в реальном времени
  * Алерты при превышении пороговых значений

*Автоматическое восстановление*::
  * Автоматический перезапуск неработающих инстансов
  * Автоматический failover для БД
  * Автоматическое масштабирование при росте нагрузки

*Изоляция*::
  * Bulkhead Pattern для критичных операций
  * Раздельные пулы ресурсов для разных типов запросов
  * Circuit Breaker для внешних зависимостей
