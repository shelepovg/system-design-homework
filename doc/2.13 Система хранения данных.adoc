=== Система хранения данных

Документ описывает выбор технологий хранения (PostgreSQL, Redis), обоснование, подходы к масштабированию и отказоустойчивости, соответствие CAP, ограничения, бэкапы и резервные системы, а также показатели хранения, рост и освобождение данных. Исходные объёмы и расчёты — документ 2.07; модель данных — документ 2.06. Схемы таблиц — документ 2.14 (ссылка на 2.06 и DBML).

==== 0. Выбор технологий и обоснование

*Принятое решение:* для хранения структурированных данных используется *только реляционная СУБД PostgreSQL*; для кэширования — *Redis*. Большие бинарные объекты (файлы пользователей, экспорты) при необходимости выносятся в объектное хранилище (S3-совместимое) или хранятся в PostgreSQL (BLOB/bytea) по политике проекта.

*Обоснование PostgreSQL:*

  * Требования к целостности (ACID), транзакциям и связям между сущностями (пользователи, запросы, платежи, аудит) — реляционная модель и SQL хорошо подходят.
  * NFR-006 (RPO = 0 для критичных данных), строгая консистентность для платежей и профилей (2.06) — PostgreSQL обеспечивает надёжную запись и репликацию.
  * Зрелая экосистема: репликация (синхронная/асинхронная), партиционирование, connection pooling, мониторинг.
  * Database per Service: каждый микросервис владеет своей БД (users_db, requests_db, payments_db, notifications_db, audit_db); единая технология упрощает эксплуатацию и найм.

*Обоснование Redis:*

  * Снижение нагрузки на БД и соблюдение latency p95 < 100 ms (NFR-002): Cache-Aside для профилей, сессий, результатов запросов; целевой cache hit rate > 80% (2.07).
  * Распределённый rate limiting (ключи в Redis), блокировки, сессии — низкая задержка доступа (p95 < 5 ms).
  * Redis Cluster даёт горизонтальное масштабирование и отказоустойчивость без смены модели данных (кэш допускает eventual consistency по TTL).

==== 0.1. Масштабирование, производительность, высокая доступность и отказоустойчивость

*PostgreSQL:*

  * *Масштабирование чтения:* read replicas (streaming replication); чтение только с реплик (~28K RPS при cache hit 80%); запись только в master (до 60K RPS). При росте нагрузки — добавление реплик в разных АЗ.
  * *Масштабирование записи:* один master на домен; при достижении лимита записи (2.07 — риск 60K RPS на master) — оптимизация (пулы, батчинг, индексы), при необходимости *шардирование* по ключу (например, user_id для user_requests) с маршрутизацией на уровне приложения.
  * *Высокая доступность:* master + минимум 2 синхронные реплики в разных АЗ; автоматический failover (Patroni, pg_auto_failover или облачный managed PostgreSQL); RTO < 1 мин, RPO = 0 при синхронной репликации.
  * *Отказоустойчивость:* health checks реплик, переключение при падении master; connection pooling (PgBouncer) для ограничения соединений и быстрого переподключения.

*Redis:*

  * *Масштабирование:* Redis Cluster (шардирование по слотам), минимум 3 master + реплики; при росте памяти и throughput — добавление узлов.
  * *Производительность:* latency p95 < 5 ms; 140K read RPS и 60–80K write/invalidate в пике (2.07) — достигаются кластером из 6+ узлов при правильном распределении ключей.
  * *Высокая доступность:* репликация master–replica; при падении master — повышение реплики до master (автоматически или через Redis Sentinel/кластер).
  * *Отказоустойчивость:* при недоступности Redis система продолжает работать с увеличенной нагрузкой на БД (деградация latency); кэш восстанавливается по TTL и запросам из БД.

*Сводка подходов:*

[options="header",cols="2,2,2"]
|===
| Компонент | Подход к масштабированию / ВД | Отказоустойчивость
| PostgreSQL (по доменам) | Read replicas; при необходимости шардирование по ключу | Master + ≥2 sync replicas, auto-failover, RPO=0
| Redis | Redis Cluster (шарды + реплики) | Репликация, failover; кэш восстанавливается из БД
|===

==== 0.2. Соответствие решению теореме CAP

*CAP:* в распределённой системе можно гарантировать не более двух из трёх свойств: Consistency (консистентность), Availability (доступность), Partition tolerance (устойчивость к разделению сети).

*Позиционирование:*

  * *Partition tolerance (P):* считается неизбежным (сеть может разделяться); проектирование идёт с учётом P.
  * *Выбор CP vs AP:* для *основного хранилища (PostgreSQL)* приоритет — *CP*: при разделении сети (partition) предпочтение отдаётся консистентности (не отдаём клиенту устаревшие или несогласованные данные); доступность восстанавливается после устранения partition за счёт failover и реплик. Синхронная репликация минимум на одну реплику обеспечивает RPO = 0 и сильную консистентность в пределах кластера.
  * *Кэш (Redis):* по сути *AP*: при partition возможна отдача устаревших данных или временная недоступность слота; данные восстанавливаются из БД и по TTL. Это допустимо, так как кэш — производный слой, а источник истины — PostgreSQL.

*Итог:* решение укладывается в CAP как *CP для данных в PostgreSQL* и *AP для Redis*; при разделении сети приложение может кратковременно деградировать по latency (чтение из БД при недоступности кэша) или по доступности записи (ожидание восстановления кворума master), что согласуется с NFR-003 (99,99%) и NFR-007 (graceful degradation).

==== 0.3. Ограничения выбранного решения

  * *Запись в PostgreSQL:* один master на домен — предел пропускной способности записи (порядка десятков тысяч RPS на инстанс после оптимизации). При дальнейшем росте требуется шардирование (сложность маршрутизации, кросс-шардовые запросы, миграции).
  * *Redis:* данные в памяти; объём ограничен размером кластера. Потеря узла до выбора новой реплики — временная потеря части ключей (восстановление по TTL и запросам к БД).
  * *Только реляционная модель:* сложные иерархии, графы или документоориентированные сценарии потребовали бы иной СУБД; в текущей модели данных (2.06) реляционная схема достаточна.
  * *Геораспределение:* синхронная репликация между регионами даёт большую задержку записи; типичная конфигурация — один регион с мульти-АЗ, при необходимости отдельный DR-регион с асинхронной репликацией (RPO > 0 для DR).

==== 0.4. Хранение и наполнение бэкапов, резервные системы

*Бэкапы PostgreSQL:*

  * *Полный бэкап:* по расписанию (например, каждые 6–12 ч) через pg_basebackup или снимки дисков (облако); хранение в отдельном хранилище (другой регион/аккаунт).
  * *Инкрементальный / WAL-архивирование:* непрерывная отправка WAL в объектное хранилище или резервный сервер для PITR (point-in-time recovery); минимальный RPO.
  * *Retention бэкапов:* не менее 30 дней (NFR, 2.06); критические домены — дольше по регуляторным требованиям.
  * *Проверка восстановления:* регулярное тестирование восстановления из бэкапа (2.10, DR).

*Резервные системы (standby):*

  * Синхронные реплики используются как hot standby для чтения и для быстрого переключения при отказе master (авто-failover).
  * При необходимости DR в другом регионе — асинхронная репликация или восстановление из WAL-архива; RTO и RPO для DR задаются отдельно (2.10).

*Redis:*

  * RDB-снимки и/или AOF по политике для восстановления после сбоя кластера; основной источник восстановления данных — наполнение из PostgreSQL при cache miss.
  * Бэкапы Redis не заменяют бэкапы БД; приоритет — целостность и бэкапы PostgreSQL.

*Сводка:*

[options="header",cols="2,2,2"]
|===
| Объект | Механизм бэкапа | Резервная система
| PostgreSQL (каждая БД) | Полный + WAL-архив, PITR | Sync replicas (hot standby), auto-failover
| Redis | RDB/AOF (опционально) | Реплики в кластере; восстановление кэша из БД
|===

==== 0.5. Обоснование применимости решения, кэши, шардирование и консистентность

===== Обоснование применимости решения

Решение «только PostgreSQL + Redis» применимо к данной системе по следующим причинам.

*Соответствие требованиям и модели данных (2.06):*

  * Данные — структурированные сущности с явными связями (пользователь → запросы, транзакции, уведомления; запрос → история, вложения). Реляционная модель и ACID обеспечивают целостность и предсказуемость.
  * Критичные операции (платежи, смена ролей, статусы запросов) требуют транзакций и строгой консистентности; NFR-006 (RPO = 0) выполняется за счёт синхронной репликации PostgreSQL.
  * Database per Service уже разбивает данные по доменам (users_db, requests_db, payments_db, notifications_db, audit_db); единая СУБД упрощает эксплуатацию, миграции и найм без потери изоляции сервисов.

*Соответствие нагрузке (2.07):*

  * Пик: 200K RPS к API, 70% GET — доминирует чтение. Масштабирование чтения через read replicas и кэш (целевой cache hit rate > 80%) укладывает нагрузку на БД в ~28K RPS на реплики и до 60K RPS записи на master.
  * Латентность: p95 < 100 ms достигается за счёт кэша (p95 < 5 ms для кэшируемых данных) и read replicas; запись в один master даёт предсказуемую задержку без распределённых коммитов.

*Итог:* выбранный стек покрывает функциональные и нефункциональные требования при текущих и прогнозируемых объёмах; необходимость в иных хранилищах (документные, графовые) по модели 2.06 не выявлена.

===== Применение кэшей

Кэш (Redis) используется по паттерну *Cache-Aside*: приложение управляет записью и инвалидацией; источник истины — PostgreSQL.

*Где и что кэшируется:*

[options="header",cols="2,2,2,2"]
|===
| Данные | Ключ (пример) | TTL | Инвалидация
| Профиль пользователя | user:{user_id} | 1 ч | При обновлении профиля (User Service)
| Сессия / токен | session:{token_hash}, user_session:{user_id} | 2 ч (сессия), 30 мин (токен) | При логауте, смене пароля, revoke
| Результат запроса | request_result:{request_id} | 24 ч | При обновлении статуса/результата запроса (Request Processing Service)
| Rate limit (счётчики) | rate_limit:{user_id}:{endpoint}, rate_limit:{ip} | 1 мин (окно) | По истечении окна (авто)
| Справочные/редко меняемые | template:{id}, config:{key} | часы–день | При публикации изменений (редко)
|===

*Правила применения:*

  * *Чтение:* сначала запрос в Redis; при miss — чтение с read replica, запись в кэш, ответ клиенту. Целевой cache hit rate > 80% для снижения нагрузки на БД и соблюдения p95 (2.07).
  * *Запись в БД:* после успешной записи/обновления в master — инвалидация соответствующих ключей в Redis (удаление или обновление), чтобы следующее чтение подтянуло актуальные данные из БД.
  * Кэш не используется как источник истины; при недоступности Redis трафик идёт в БД (деградация по latency, без потери корректности).

*Ограничения:* возможна кратковременная несогласованность между кэшем и БД в окне между записью в БД и инвалидацией (миллисекунды); для критичных операций (например, баланс, статус платежа) чтение после записи можно выполнять с master или с реплики с учётом репликационного лага. Для профилей и результатов запросов такое окно допустимо (2.13, CAP: кэш — AP).

===== Допустимость шардирования

Шардирование допустимо как *расширение* при исчерпании возможностей одного master по записи или при росте объёма данных одного домена сверх разумного предела одного кластера.

*Когда рассматривать шардирование:*

  * *Запись:* при устойчивой нагрузке записи на master значительно выше расчётных ~60K RPS после оптимизации (пулы, батчинг, индексы) или при необходимости масштабирования записи на порядок (2.07, 2.08).
  * *Объём:* при росте одной БД (например, requests_db) до объёмов, при которых управление одним кластером (бэкапы, миграции, failover) становится риском или операционной нагрузкой.

*По каким ключам допустимо шардировать:*

  * *users_db:* по `user_id` — запросы пользователя привязаны к нему; кросс-шардовые запросы (например, «все пользователи с ролью X») редки и могут выполняться через отдельный слой (агрегация, поиск).
  * *requests_db:* по `user_id` (запросы пользователя) или по диапазону `request_id`/времени — типичные сценарии «мои запросы» не требуют объединения шардов.
  * *payments_db, notifications_db:* по `user_id` или по идентификатору транзакции/уведомления с учётом того, что агрегации «все платежи за период» потребуют сбора с шардов или отдельного хранилища отчётов.
  * *audit_db:* по времени (партиционирование по дате уже предусмотрено в 2.13); при необходимости — шардирование партиций по диапазонам дат.

*Ограничения и риски:*

  * Транзакции и FK действуют только в пределах одного шарда. Кросс-шардовые транзакции не поддерживаются нативно; координация (Saga, outbox) остаётся на уровне приложения.
  * Запросы, требующие объединения данных с нескольких шардов (например, отчёты по всем пользователям), выполняются через агрегацию в приложении или выносятся в отдельный аналитический слой (ETL, read-only копия).
  * Миграция на шардирование — дорогостоящее изменение; допустимо только при обоснованной необходимости по нагрузке или объёму.

*Итог:* шардирование допустимо и технически возможно по `user_id` и по времени/диапазонам; применяется по мере роста, а не по умолчанию.

===== Отсутствие неконсистентности (обеспечение консистентности)

В выбранном решении неконсистентность минимизируется за счёт следующих мер.

*PostgreSQL — единый источник истины, сильная консистентность:*

  * *Одна точка записи (master) на домен:* все записи и обновления идут в master; нет конфликтующих источников записи (multi-master не используется).
  * *Синхронная репликация* минимум на одну реплику перед подтверждением записи клиенту: обеспечивает RPO = 0 и гарантию, что при failover данные не потеряются; реплики содержат согласованное состояние.
  * *ACID и транзакции:* все связанные изменения в рамках одной БД выполняются в одной транзакции (например, payment_transactions + payment_outbox); откат при ошибке сохраняет целостность.

*Кэш (Redis) — производный слой:*

  * Кэш не является источником истины; при расхождении приоритет у PostgreSQL. Инвалидация при записи (Cache-Aside) ограничивает окно возможной устарелости данными в кэше.
  * Короткие TTL снижают риск длительного отображения устаревших данных при сбое инвалидации.
  * Для сценариев «прочитал после записи» (read-after-write) при необходимости строгой свежести — чтение с master или с реплики с учётом известного лага репликации.

*Идемпотентность и дедупликация:*

  * Платежи — по idempotency key; повторная доставка сообщений из Kafka обрабатывается идемпотентно (дедупликация по message id / ключу). Это исключает двойное списание и дублирование побочных эффектов.

*Итог:* архитектура обеспечивает сильную консистентность данных в PostgreSQL и ограничивает допустимую «мягкую» несогласованность только кэшем с коротким окном и явной инвалидацией; требования NFR-006 (RPO = 0) и целостность критичных данных соблюдаются.

==== 0.6. Геораспределённость, standby, бэкапы и распределённая обработка

===== Уточнение необходимости в геораспределённости

*Базовый вариант (рекомендуемый для MVP и типовой эксплуатации):*

  * *Один регион, мульти-АЗ (2–3 зоны доступности):* вся активная обработка и хранение в одном регионе; инстансы сервисов, PostgreSQL master и синхронные реплики, Redis Cluster и Kafka размещены в разных АЗ. Этого достаточно для NFR-003 (99,99%), NFR-006 (RPO = 0), NFR-007 (отказоустойчивость при отказе АЗ/узла).
  * *Геораспределённость в смысле «активная запись/чтение в нескольких регионах» не обязательна:* синхронная репликация между регионами даёт высокую задержку записи и операционную сложность; для целевой нагрузки (2.07) и латентности (p95 < 100 ms) один регион с мульти-АЗ достаточен.

*Когда нужна геораспределённость:*

  * *DR (катастрофа региона):* отдельный *резервный регион* для аварийного восстановления (2.10). Данные туда доводятся *асинхронной* репликацией или восстановлением из бэкапов; RPO для DR может быть > 0 (минуты). Переключение на DR — вручную или по процедуре при потере основного региона; RTO DR задаётся отдельно (например, < 1 ч по 2.10).
  * *Бэкапы в нескольких регионах:* NFR-006 и 2.10 требуют хранение копий бэкапов минимум в 2 регионах — это *геораспределённость хранения бэкапов*, а не активной БД.
  * *Активная обработка в нескольких регионах (multi-region active):* только при явном требовании — низкая латентность для пользователей в разных частях мира или регуляторное требование к размещению данных в нескольких регионах. Тогда рассматриваются отдельные кластеры по регионам с асинхронной синхронизацией и маршрутизацией трафика (значительное усложнение архитектуры и не принимо к нашему решению).

*Итог:* необходимость в геораспределённости для *активной* работы в нескольких регионах не предполагается по умолчанию; обязательны мульти-АЗ в одном регионе и геораспределённость бэкапов (и при необходимости — DR standby в другом регионе).

===== Standby (резервные инстансы)

*Hot standby в пределах региона (обязательно):*

  * *PostgreSQL:* синхронные реплики (минимум 2) являются hot standby — принимают поток репликации, обслуживают чтение, при отказе master одна из реплик повышается до master (авто-failover). RTO < 1 мин, RPO = 0 (2.08, 2.10).
  * *Redis:* реплики в кластере — standby для своих master-узлов; при падении master реплика повышается до master (автоматически или через Sentinel/кластер).
  * *Сервисы приложения:* минимум 3 инстанса в разных АЗ; при падении одного остальные продолжают обслуживать трафик (standby по сути встроен в пул за LB).

*DR standby в другом регионе (опционально, по требованию DR):*

  * *PostgreSQL:* асинхронная репликация в второй регион или периодическое восстановление из WAL-архива/полного бэкапа. В момент катастрофы основного региона реплика в DR-регионе может быть повышена до master; возможна потеря последних секунд/минут данных (RPO > 0 для DR). Детали сценариев — 2.10.
  * *Redis:* в DR-регионе кэш заполняется заново при переключении трафика (источник истины — PostgreSQL); отдельный standby Redis в DR не обязателен.
  * *Kafka, IAM, приложение:* развёртывание в DR-регионе по плану DR (образы, конфигурация); «холодный» или «тёплый» standby в зависимости от целевого RTO.

*Сводка:*

[options="header",cols="2,2,2"]
|===
| Компонент | Standby в регионе | Standby в DR-регионе
| PostgreSQL | Hot standby (sync replicas), auto-failover | Опционально: async replica или restore из бэкапа
| Redis | Реплики в кластере | Восстановление из БД при переключении
| Сервисы, LB, API Gateway | Пул инстансов в нескольких АЗ | Развёртывание по плану DR (2.10)
|===

===== Бэкапы (краткая сводка)

Полное описание — п. 0.4 выше и документ 2.10 (DR, проверка восстановления).

*Обязательные параметры:*

  * *PostgreSQL:* полный бэкап по расписанию (например, каждые 6–12 ч); непрерывная или почасовная отправка WAL в отдельное хранилище для PITR. Хранение бэкапов: минимум 30 дней; копии минимум в 2 регионах (NFR-006, 2.10).
  * *Проверка восстановления:* ежемесячное тестирование восстановления из бэкапа (2.10); результаты фиксируются, runbook обновляются.
  * *Redis:* бэкапы RDB/AOF опциональны; приоритет — целостность PostgreSQL; кэш восстанавливается из БД при cache miss.

*Наполнение и использование:* бэкапы наполняются автоматически (pg_basebackup, WAL-архив, снимки дисков или облака); хранятся в объектном хранилище (другой регион/аккаунт). Используются для PITR при сбое, для восстановления в DR-регионе и для проверки процедур DR.

===== Распределённая обработка

*Что понимается под распределённой обработкой в данном решении:*

  * *Масштабируемые приложения:* микросервисы (User, Request Processing, Payment, Notification, Audit) работают в нескольких инстансах за Load Balancer; запросы распределяются по инстансам без единой точки отказа (горизонтальное масштабирование, NFR-004).
  * *Распределённое чтение данных:* чтение из PostgreSQL — только с read replicas (несколько реплик в разных АЗ); чтение из Redis — через Redis Cluster (шардирование по слотам, реплики). Запись — централизованно в master БД по домену.
  * *Асинхронная распределённая обработка через Event Bus:* тяжёлые или некритичные по латентности сценарии (уведомления, аудит, часть интеграций) выполняются потребителями Kafka в нескольких инстансах; партиции топиков распределены по брокерам, потребители масштабируются независимо (2.08).
  * *Очередь запросов (Request Processing Service):* воркеры обрабатывают задачи из очереди (Kafka или внутренняя очередь); несколько воркеров на нескольких инстансах — распределённая обработка без единого координатора (приоритизация и FIFO по политике сервиса).

*Чего нет в базовом решении:*

  * Нет единого «распределённого вычислительного кластера» (типа Spark/Flink) для произвольной аналитики или пакетной обработки больших объёмов — такая потребность выносится в отдельный аналитический слой или бэклог.
  * Нет распределённых транзакций между несколькими БД: каждая БД принадлежит одному сервису; координация между сервисами — через события (Kafka), Saga, outbox (2.06, 2.08).

*Итог:* распределённая обработка обеспечивается масштабируемыми микросервисами, read replicas, Redis Cluster, Kafka и воркерами очереди запросов; дополнительная распределённая вычислительная платформа не предполагается без отдельного требования.

==== 1. Показатели по хранению данных

Сводка объёмов по хранилищам (типичный горячий слой и оценка роста в год). Детализация расчётов — 2.07, п. 1.2.

|===

| requests_db | 500 ГБ – 2 ТБ | 500 ГБ – 2 ТБ | Request Processing Service (запросы, история, вложения — метаданные)
| payments_db | 20–100 ГБ | 5–50 ГБ | Payment Service (транзакции, outbox)
| notifications_db | 50–200 ГБ | 0,5–2 ТБ | Notification Service (история доставки, шаблоны)

| Объектное хранилище | 1–5 ПБ (суммарно) | 100 ТБ – 1 ПБ | Файлы, артефакты, отчёты (FR-011: 10 ГБ/пользователь, 1000 ресурсов)

*Метрики для планирования:* объём горячего слоя определяет размер дисков и бэкапов; рост/год — планирование расширения и политик архивирования.

==== 2. Рост данных

*Источники роста:*

  * *users_db:* регистрация новых пользователей, накопление истории сессий (ограничено retention).
  * *requests_db:* основной драйвер — 2 млн активных × 20 запросов/день × 365 дней; вложения и результаты — в объектном хранилище.
  * *payments_db, notifications_db:* пропорционально транзакциям и доставкам.
  * *audit_db:* объём записей при sampling (например, 1 из 100 запросов) и события безопасности.
  * *Объектное хранилище:* загрузка файлов пользователями, экспорты, отчёты; лимит на пользователя 10 ГБ (FR-011).

*Тренды:* при росте числа активных пользователей (NFR-001, до 50% в первый год) рост объёмов масштабируется; requests_db и объектное хранилище дают наибольший прирост в абсолютных значениях.

==== 3. Освобождение данных

Политики освобождения места и соответствие регуляторным/операционным требованиям.

[options="header",cols="2,2,2"]
|===
| Хранилище / данные | Механизм освобождения | Retention / примечание
| user_sessions | Удаление/архив по дате expires_at или revoked_at | Активные сессии; старые — purge (например, > 30 дней)
| user_requests, request_history, request_attachments | Архивация в cold, затем удаление | Hot: последние 12 мес; warm: 1–3 года; cold/удаление по политике (например, > 5 лет)
| payment_*, notifications | Юридический retention (например, 5–7 лет) | Удаление после истечения срока хранения; архив в cold перед удалением
| audit_logs, security_events | Партиционирование по дате; архив партиций в cold; purge старых партиций | Hot: 30–90 дней; warm: до 1 года; cold/удаление по политике (например, > 7 лет)
| notification_templates | Практически не удаляются | Статичные/редко меняемые
| Redis | TTL по ключам (user 1 ч, session 2 ч, token 30 мин, request 24 ч, rate_limit 1 мин) | Автоматическое освобождение по истечении TTL
| Kafka | Retention топиков (часы–дни) | Освобождение после потребления и истечения retention
| Объектное хранилище | Lifecycle: переход в cold (например, Glacier), удаление после срока | Отчёты/экспорты: например, 30 дней; пользовательские файлы — по политике и лимитам
|===

*Итог:* освобождение за счёт архивирования в холодное хранилище (warm/cold), purge по retention и TTL в кэше и очередях. Критичные данные (платежи, аудит) — удаление только после истечения установленного срока хранения.

==== 4. Горячие, тёплые и холодные данные

[options="header",cols="2,2,2,2"]
|===
| Домен / данные | Hot (быстрый доступ) | Warm (архив, реже запросы) | Cold (долгосрочный архив, редкий доступ)
| users_db | Активные профили, роли, текущие сессии | — | Старые сессии (перед purge)
| requests_db | Запросы за последние 12 мес, активные обработки | 1–3 года (поиск, отчёты) | > 3–5 лет (архив, восстановление по запросу)
| payments_db, notifications_db | Текущие транзакции, недавние уведомления | История за несколько лет (аудит, отчёты) | После истечения юридического retention
| audit_db | Последние 30–90 дней (расследования, алерты) | До 1 года | Старые партиции в object storage / холодном хранилище
| Объектное хранилище | Часто запрашиваемые файлы, свежие экспорты | Реже запрашиваемые | Lifecycle в Glacier/аналог; долгосрочное хранение
| Redis | Все ключи кэша (по TTL) | — | —
|===

*Реализация:* hot — PostgreSQL (SSD, индексы), Redis; warm — те же БД с партициями или отдельные тома с более дешёвым хранилищем; cold — объектное хранилище с холодным tier, архивные форматы.

==== 5. Статичные и модифицируемые данные

[options="header",cols="2,2,2"]
|===
| Данные | Характер | Нагрузка на изменение
| users (профиль), user_roles, user_permissions | Модифицируемые (редко) | Низкая: обновление профиля, смена ролей
| user_sessions | Часто меняемые (создание/завершение) | Высокая при логине/логауте
| user_requests, request_history, request_attachments | Запись при создании; обновление статуса и результата | Запись/обновление в пике до десятков тысяч RPS (2.07)
| payment_transactions, payment_history, payment_outbox | Транзакции: запись и обновление статуса | Критично по целостности; запись и редкие обновления
| notifications, notification_templates | Уведомления: запись и обновление статуса доставки; шаблоны — почти статичные | Средняя на запись; шаблоны — редкие изменения
| audit_logs, security_events | Добавление (append-only) | Только вставка, без обновления
| system_metrics, health_checks, alerts | Метрики: append; алерты: обновление статуса | Append-нагрузка; обновление при acknowledge/resolve
| Redis (кэш) | Временные, перезаписываемые | Высокая: запись и инвалидация при cache-aside
|===

*Итог:* большая часть операционной нагрузки на запись приходится на user_requests, user_sessions, payment/notification статусы, outbox и кэш; аудит и метрики — в основном append-only. Статичные/редко меняемые: справочники, шаблоны, завершённые транзакции (после фиксации статуса).

==== 6. Нагрузка на чтение и обновление данных

Оценки при пиковой нагрузке 200 000 RPS к API Gateway (2.07). Доля GET 70%, POST/PUT 25%, DELETE 5%.

*PostgreSQL (суммарно по всем БД):*
  * *Чтение:* при cache hit rate 80% на read replicas приходится порядка *28 000 RPS* (20% от 140 000 GET).
  * *Запись/обновление:* до *60 000 RPS* на master (POST/PUT/DELETE и служебные записи: outbox, история, статусы).

*Redis (кэш):*
  * *Чтение:* до *140 000 RPS* (все GET проходят через Cache-Aside).
  * *Запись/инвалидация:* порядка *60 000–80 000* операций/с в пике (запись в кэш при miss, инвалидация при обновлении данных).

*Распределение по доменам (ориентировочно):*
  * Наибольшая доля чтения и записи — *requests_db* (запросы, история) и кэш результатов запросов.
  * *users_db* и кэш профилей/сессий — значимое чтение, запись при логине/обновлении профиля.
  * *payments_db*, *notifications_db* — меньше RPS, но критичны по целостности при записи.
  * *audit_db* — в основном пакетная запись (consumers из Kafka), минимальное чтение в реальном времени.

[options="header",cols="2,2,2"]
|===
| Компонент | Чтение (RPS, пик) | Запись/обновление (RPS, пик)
| PostgreSQL read replicas | ~28 000 | —
| PostgreSQL master | — | до ~60 000
| Redis (кэш) | до ~140 000 | ~60 000 – 80 000
|===

Учёт данной нагрузки необходим для проектирования пулов соединений, количества реплик, размера кластера Redis и политик кэширования (2.07, 2.08).
