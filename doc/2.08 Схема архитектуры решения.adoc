=== Схема архитектуры решения

Концептуальная схема архитектуры с детализацией по требованиям высокой доступности (NFR-003), отказоустойчивости (NFR-007), масштабируемости (NFR-004) и производительности (NFR-002, расчёты в документе 2.07). Целевые показатели: 99,99% SLA, 200K RPS пик, p95 latency < 100 ms, RTO < 1 мин, RPO = 0 для критичных данных.

==== Целевые показатели (сводка по 2.07)

[options="header",cols="2,2"]
|===
| Показатель | Значение
| Throughput (пик / средняя / при деградации) | 200K / 50K / 20K RPS
| Latency p95 (end-to-end) | < 100 ms
| Доступность | 99,99%
| RTO (критичные сервисы) | < 1 мин
| RPO (критичные данные) | 0
| Минимум инстансов (критичные компоненты) | 3, в разных АЗ
| Масштабирование | до 10x без потери производительности
|===

==== Высокая доступность

*Принципы:* мульти-АЗ (минимум 2–3 зоны доступности в регионе), активный-активный или активный-резерв по компонентам, автоматический failover в пределах RTO, health checks каждые 10 с.

*Load Balancer*::
  * Режим: активный-активный (минимум 2 инстанса в разных АЗ).
  * Health checks: каждые 10 с, исключение неработающих бэкендов из пула.
  * Роль: единая точка входа для 200K RPS, TLS termination, маршрутизация на Front и API Gateway.

*API Gateway*::
  * Минимум 3 инстанса в разных АЗ; для 200K RPS — расчётное значение не менее 15–20 инстансов в пике (см. 2.07).
  * Health: liveness/readiness каждые 10 с; graceful shutdown (30 с на завершение запросов).
  * Кэш JWT обязателен: снижение обращений к IAM до 5–10% RPS, соблюдение бюджета latency (≤10 ms на проверку токена).

*Backend Services (микросервисы)*::
  * Минимум 3 инстанса на сервис в разных АЗ; Request Processing Service — наибольшее число инстансов (ориентир 2–5K RPS на инстанс, автоскейлинг по CPU 70%, latency p95 150 ms, queue depth 1000).
  * Database per Service; каждый сервис — отдельный пул и масштабирование.
  * Health checks каждые 10 с; graceful shutdown.

*Database (PostgreSQL)*::
  * Master + минимум 2 синхронные реплики (в разных АЗ); автоматический failover, RTO < 1 мин, RPO = 0 для критичных данных.
  * Чтение: только через read replicas (нагрузка чтения ~28K RPS при cache hit rate 80%); запись только в master (до 60K RPS — см. 2.07).
  * Connection pooling (PgBouncer или аналог), запас пула не менее 20%.

*Cache (Redis)*::
  * Redis Cluster, минимум 3 узла (лучше 6 для N реплик); репликация, автоматический failover.
  * Целевой cache hit rate > 80%; latency p95 < 5 ms; иначе рост нагрузки на БД и деградация p95.

*IAM (Keycloak)*::
  * Минимум 2 инстанса в разных АЗ; кэширование валидных токенов и JWKS на стороне API Gateway для снижения нагрузки.

*Event Bus (Kafka)*::
  * Кластер с репликацией партиций (min.insync.replicas ≥ 2); брокеры в разных АЗ; при отказе брокера — переизбрание leader без потери сообщений.

==== Отказоустойчивость

*Circuit Breaker* — для всех вызовов к внешним системам (платежи, уведомления, внешний IAM): при превышении порога ошибок или таймаутов — разрыв цепи, быстрый отказ, повторное подключение по таймеру/полуручному сбросу.

*Retry* — для временных сбоев (сеть, 5xx): ограниченное число попыток, exponential backoff; операции идемпотентны где необходимо (платежи — по idempotency key).

*Bulkhead* — изоляция пулов потоков/соединений: критичные операции (обработка запросов, платежи) не блокируются длительными или перегруженными фоновыми задачами (аудит, уведомления).

*Graceful degradation*: при недоступности некритичного компонента (например, часть уведомлений) — основная функциональность (запросы, чтение/запись, платежи) сохраняется; при перегрузке — приоритизация по очереди запросов (приоритет 1–2 в первую очередь), возможное ограничение низкоприоритетных операций.

*Health checks и исключение из пула*: liveness/readiness каждые 10 с; при падении инстанса load balancer и оркестратор исключают его из приёма трафика в пределах десятков секунд.

==== Размещение Rate Limiter, Retry, Circuit Breaker

Ниже — где в архитектуре размещать ограничение частоты запросов, повторные попытки и разрыв цепи при сбоях зависимостей. Детали по лимитам — NFR-005 (1.02), по обработке ошибок — NFR-007 (1.02), по инцидентам — 2.10.

*Rate Limiter (ограничение частоты запросов)*

[options="header",cols="2,2,4"]
|===
| Место | Уровень / ключ | Назначение и параметры
| API Gateway | По пользователю (user_id / JWT sub) | Лимит на пользователя: до 1000 запросов/мин (NFR-005); защита от злоупотребления и перегрузки отдельным клиентом.
| API Gateway | По IP (анонимные запросы: логин, health) | До 5000 запросов/мин с IP; защита от DDoS и brute-force на /auth/login (доп. лимит на логин — см. 1.01).
| API Gateway | По endpoint (опционально) | Разные лимиты для тяжёлых операций (например, экспорт, отчёты) vs лёгких (GET профиля); снижение пиков на критичных путях.
| Load Balancer (опционально) | По IP / глобально | Доп. слой защиты от флуда; при необходимости — только базовый глобальный cap, детализация в Gateway.
|===

*Реализация:* Token Bucket или Sliding Window; для распределённого лимита — Redis (ключи вида `rate_limit:{user_id}:{endpoint}` или `rate_limit:{ip}`), TTL по окну. При превышении: HTTP 429, заголовок `Retry-After`. Связь с моделью данных (ключи кэша) — 2.06.

*Retry (повторные попытки)*

[options="header",cols="2,2,4"]
|===
| Место | Направление вызова | Условия и параметры
| API Gateway | Gateway → Backend (микросервис) | При 5xx и таймауте: ограниченное число попыток (например, 2–3), exponential backoff; таймаут запроса к Backend меньше таймаута клиента; идемпотентность на стороне Backend для безопасного retry (X-Idempotency-Key для мутирующих операций при необходимости).
| API Gateway | Gateway → IAM (внутр./внеш.) | При временной недоступности (5xx, timeout): 1–2 повтора с коротким backoff; не задерживать все запросы — при повторном сбое быстрый отказ (см. Circuit Breaker).
| Payment Service | Backend → Платёжная система (внешняя) | Retry только для идемпотентных операций (idempotency key); ограниченное число попыток (например, 3), exponential backoff; не retry для неидемпотентных или при 4xx от провайдера.
| Notification Service | Backend → Внешняя система уведомлений | Retry при 5xx/timeout; ограниченное число попыток; backoff; идемпотентность обработки сообщений из Kafka (дедупликация по message id / ключу).
| Все микросервисы | Backend → PostgreSQL / Redis | Обычно не retry на уровне приложения для каждого запроса — управление через connection pool и таймауты; при транзиентных сбоях соединения — переподключение пула; для явного retry (например, deadlock) — ограниченно и с backoff.
|===

*Circuit Breaker (разрыв цепи)*

[options="header",cols="2,2,4"]
|===
| Место | Зависимость | Назначение и поведение
| API Gateway | Gateway → IAM (внутренний Keycloak) | При росте доли ошибок/таймаутов (например, > 50% за окно 10–30 с) — открытие цепи: не вызывать IAM, отвечать 503 или использовать кэш JWT «до последнего известного состояния» с предупреждением в метриках; полуручной или таймерный (например, 30 с) пробный вызов для восстановления.
| API Gateway | Gateway → IAM внешний | Аналогично: при недоступности внешнего IAM — быстрый отказ (503), не накапливать очередь запросов на логин; алерт, retry по таймеру.
| API Gateway | Gateway → Backend (по маршруту/сервису) | Опционально: при массовых 5xx от конкретного микросервиса — разрыв цепи на маршрут, чтобы не перегружать падающий сервис; возврат 503; восстановление по таймеру или health check.
| Payment Service | Backend → Платёжная система | Обязательно (2.08, NFR-007): при недоступности или > N% ошибок — открытие цепи; не вызывать провайдера; сохранять транзакции в Pending, публиковать в outbox; повторная отправка/retry после восстановления цепи; алерт и runbook «Платежи» (2.10).
| Notification Service | Backend → Внешняя система уведомлений | При недоступности провайдера — разрыв цепи; сообщения остаются в Kafka (или в локальной очереди); доставка после восстановления; избежание каскадных задержек и исчерпания потоков.
| User Service | User Service → IAM (синхронизация) | При сбоях IAM — не блокировать основные операции (чтение профиля из БД/кэша); синхронизацию отложить, retry/CB по политике.
|===

*Сводка по компонентам*

[options="header",cols="2,2,2,2"]
|===
| Компонент | Rate Limiter | Retry | Circuit Breaker
| Load Balancer | Опционально (глобально по IP) | — | —
| API Gateway | Да (по user, IP, опционально по endpoint) | Да (→ Backend, → IAM) | Да (→ IAM внутр./внеш., опционально → Backend)
| Payment Service | — | Да (→ платёжный провайдер, идемпотентно) | Да (→ платёжный провайдер)
| Notification Service | — | Да (→ провайдер уведомлений, при потреблении Kafka) | Да (→ провайдер уведомлений)
| Request Processing Service | — | По необходимости к внешним вызовам | По внешним зависимостям при наличии
| User Service | — | При вызовах IAM (синхронизация) | Опционально (→ IAM при синхронизации)
|===

Мониторинг: метрики по каждому Circuit Breaker (состояние open/half-open/closed, количество отказов), по Rate Limiter (количество 429, топ превышений по user/IP), по Retry (количество повторов, успех после N-й попытки) — см. 2.09, 2.10.

==== Масштабирование

*Горизонтальное масштабирование* — основной способ; вертикальное — только для специфичных случаев (например, master БД до исчерпания вариантов оптимизации).

*Триггеры scale-out (по NFR-004):* CPU > 70% или Memory > 80% в течение 5 мин; latency p95 > 150 ms в течение 3 мин; queue depth > 1000; error rate > 0,5% в течение 5 мин.

*Триггеры scale-in:* CPU < 30%, Memory < 50%, latency p95 < 50 ms, queue depth < 100 в течение 15 мин. Минимум инстансов не опускается ниже 3 для критичных сервисов. Cooldown между операциями масштабирования: 5 мин; шаг: ±1–2 инстанса.

*Компоненты и ориентиры масштабирования (по 2.07):*
  * API Gateway: min 3, в пике 15–20+ инстансов (10–15K RPS на инстанс).
  * Request Processing Service: min 3, масштабирование по нагрузке (2–5K RPS на инстанс), по queue depth и latency.
  * Остальные Backend-сервисы: min 3, масштабирование по метрикам и объёму операций.
  * Database: добавление read replicas при росте нагрузки чтения; при росте записи — оптимизация, пулы, при необходимости шардирование (см. 2.07).
  * Redis Cluster: добавление узлов при росте памяти и throughput (140K+ read RPS, инвалидации).
  * Система мониторинга: масштабирование коллекторов и хранилищ при росте объёма метрик и логов.

*Zero-downtime deployment:* Blue-Green или Canary; постепенный перевод трафика; автоматический rollback при росте ошибок или деградации latency.

==== Производительность (критический путь и бюджет задержки)

Бюджет задержки для p95 < 100 ms (типовой GET, по 2.07): API Gateway ≤5 ms, IAM/JWT ≤10 ms, Backend ≤15 ms, Cache ≤5 ms, Database (при cache miss) ≤20 ms, сеть ≤5 ms, резерв ≤40 ms.

*Реализация:*
  * *Кэш (Redis):* Cache-Aside для чтения; Write-Through для критичных данных; TTL и инвалидация по событиям; целевой hit rate > 80%.
  * *Чтение БД:* только через read replicas; запросы с индексами, целевой p95 запросов < 20 ms.
  * *Запись БД:* master, connection pooling, без длительных транзакций на критичном пути; батчинг и асинхронная запись для аудита/логов через Event Bus.
  * *JWT:* кэш в API Gateway (или отдельном слое), локальная валидация RS256 с кэшем JWKS — снижение вызовов IAM до 5–10% RPS.
  * *Очередь запросов (Request Processing Service):* приоритизация (1–10), при перегрузке — обслуживание высокоприоритетных; мониторинг queue depth (алерт при > 1000).

==== Схема размещения и связей (концепт)

[options="header",cols="2,4"]
|===
| Уровень | Компоненты и размещение
| Вход | LB (≥2 АЗ) → Front (CDN/статику), API Gateway (≥3 АЗ, 15–20+ в пике)
| Слой приложения | Backend Services: min 3 по АЗ; Request Processing Service масштабируется наиболее активно
| Данные | PostgreSQL: 1 master + ≥2 sync replicas (разные АЗ); read — только replicas; Redis Cluster ≥3 узла (лучше 6)
| Асинхронность | Event Bus (Kafka): кластер, репликация; потребители — Notification, Payment, Audit
| Идентификация | IAM (≥2 АЗ); кэш JWT и JWKS на API Gateway
| Мониторинг | Сбор логов (Kafka) и метрик (Prometheus); алерты по latency, error rate, hit rate, queue depth
|===

Визуальные схемы контекста и контейнеров — в LikeC4 (контекстная диаграмма, детализация системы мониторинга, пользователи и внешние системы) и при необходимости в документе 4.01 (схема развёртывания).
